Esta versão da API introduz mudanças incompatíveis com versões anteriores. Siga este guia para atualizar sua integração.

**Data de vigência:** 21 de abril de 2026

---

## Resumo das mudanças

| Área | Mudança | Tipo |
| ---- | ------- | ---- |
| Formato do `message_id` | Agora prefixado com `msg_` | Breaking |
| `request_id` no corpo | Removido — use o header `X-Request-ID` | Breaking |
| `GET /messages/{request_id}/status` | Removido — use `GET /messages/{message_id}/status` | Breaking |
| Webhook `message.received` | Campo `content` removido; adicionados `preview` e `content_type` | Breaking |
| Webhook `message.edited` | Campo `new_text` removido; adicionado `new_preview` | Breaking |
| Typing indicator | Campo `value` renomeado para `is_typing` | Breaking |
| `POST /numbers` | Campo `phone` renomeado para `phone_number` | Breaking |
| `GET /numbers` | Shape da resposta mudou para Stripe-like | Breaking |

---

## 1. Formato do `message_id`

O identificador de mensagem agora inclui o prefixo `msg_` seguido de 32 caracteres hexadecimais minúsculos.

**Antes:**

```json
{
  "message_id": "01993c5a1b24d3e9f01a2b3c4d5e6f70"
}
```

**Depois:**

```json
{
  "message_id": "msg_01993c5a1b24d3e9f01a2b3c4d5e6f70"
}
```

**Validação:** `^msg_[0-9a-f]{32}$`

**O que fazer:** Atualize qualquer código que armazene ou compare `message_id` para incluir o prefixo `msg_`. Atualize validações de formato nos seus sistemas.

---

## 2. Correlação de requisição: `request_id` removido do corpo

O campo `request_id` foi removido do corpo das respostas. A correlação de requisições agora é feita exclusivamente pelo header `X-Request-ID`.

**Antes:**

```json
{
  "message_id": "msg_01993c5a1b24d3e9f01a2b3c4d5e6f70",
  "request_id": "req_01993c5a1b24d3e9f01a2b3c4d5e6f70",
  "status": { "state": "pending", "at": "2026-04-21T10:00:00Z" }
}
```

**Depois:**

```http
202 Accepted
X-Request-ID: req_01993c5a1b24d3e9f01a2b3c4d5e6f70
```

```json
{
  "message_id": "msg_01993c5a1b24d3e9f01a2b3c4d5e6f70",
  "status": { "state": "pending", "at": "2026-04-21T10:00:00Z" }
}
```

**O que fazer:** Leia o `X-Request-ID` do header da resposta. Para rastrear uma requisição específica nos eventos, use `GET /v1/events?request_id=req_...`.

---

## 3. Consulta de status por `request_id` removida

O endpoint `GET /messages/{request_id}/status` foi removido. Use o `message_id` retornado na resposta de envio.

**Antes:**

```bash
curl "https://api.turbonotify.com/messages/req_abc123/status" \
  -H "Authorization: Bearer SUA_CHAVE"
```

**Depois:**

```bash
curl "https://api.turbonotify.com/messages/msg_01993c5a1b24d3e9f01a2b3c4d5e6f70/status" \
  -H "Authorization: Bearer SUA_CHAVE"
```

**O que fazer:** Armazene o `message_id` (não o `request_id`) para consultas de status futuras. Para correlacionar requisições com eventos, use a API de eventos com `request_id` no query parameter.

---

## 4. Webhook `message.received`: `content` removido

O campo `content` foi removido do payload do evento `message.received`. Em seu lugar, foram adicionados `preview` e `content_type`.

**Antes:**

```json
{
  "id": "evt_abc123",
  "type": "message.received",
  "timestamp": "2026-04-21T10:00:00Z",
  "data": {
    "message_id": "01993c5a1b24d3e9f01a2b3c4d5e6f70",
    "from": "+5511999999999",
    "to": "+5511973117056",
    "content": "Olá, gostaria de saber sobre o pedido",
    "received_at": "2026-04-21T10:00:00Z"
  }
}
```

**Depois:**

```json
{
  "id": "evt_abc123",
  "type": "message.received",
  "timestamp": "2026-04-21T10:00:00Z",
  "data": {
    "message_id": "msg_01993c5a1b24d3e9f01a2b3c4d5e6f70",
    "from": { "number": "+5511999999999" },
    "to": { "number": "+5511973117056" },
    "content_type": "text",
    "preview": "Olá, gostaria de saber sobre o pedido",
    "received_at": "2026-04-21T10:00:00Z"
  }
}
```

**O que fazer:**
- Substitua o uso de `data.content` por `data.preview` para exibição ou logging rápido.
- Para o corpo completo da mensagem, chame `GET /v1/messages/{message_id}` dentro da janela de retenção.
- Use `data.content_type` para identificar o tipo de conteúdo recebido.
- Atualize referências a `data.from` e `data.to`: agora são objetos `{number: string}`, não strings planas.

---

## 5. Webhook `message.edited`: `new_text` removido

O campo `new_text` foi removido do payload do evento `message.edited`. Em seu lugar foi adicionado `new_preview`.

**Antes:**

```json
{
  "id": "evt_abc124",
  "type": "message.edited",
  "timestamp": "2026-04-21T10:05:00Z",
  "data": {
    "message_id": "01993c5a1b24d3e9f01a2b3c4d5e6f70",
    "new_text": "Texto corrigido da mensagem completa.",
    "edited_at": "2026-04-21T10:05:00Z"
  }
}
```

**Depois:**

```json
{
  "id": "evt_abc124",
  "type": "message.edited",
  "timestamp": "2026-04-21T10:05:00Z",
  "data": {
    "message_id": "msg_01993c5a1b24d3e9f01a2b3c4d5e6f70",
    "new_preview": "Texto corrigido da mensagem co",
    "edited_at": "2026-04-21T10:05:00Z"
  }
}
```

**O que fazer:** Substitua o uso de `data.new_text` por `data.new_preview`. O campo `new_preview` é um recorte permanente de até 50 grafemas do novo conteúdo. Para o corpo completo atualizado, chame `GET /v1/messages/{message_id}`.

---

## 6. Typing indicator: `value` renomeado para `is_typing`

O campo `value` no payload do typing indicator foi renomeado para `is_typing`.

**Antes:**

```json
{
  "to": "+5511999999999",
  "value": true
}
```

**Depois:**

```json
{
  "to": "+5511999999999",
  "is_typing": true
}
```

**O que fazer:** Atualize as chamadas a `POST /v1/typing-indicator` para usar `is_typing` no lugar de `value`.

---

## 7. `POST /numbers`: campo `phone` renomeado para `phone_number`

O campo do número de telefone ao criar um número extra foi renomeado.

**Antes:**

```json
{
  "alias": "vendas-sp",
  "phone": "+5511988887777"
}
```

**Depois:**

```json
{
  "alias": "vendas-sp",
  "phone_number": "+5511988887777"
}
```

**O que fazer:** Atualize qualquer chamada a `POST /v1/numbers` para usar `phone_number` no campo do telefone.

---

## 8. `GET /numbers`: novo formato de resposta

A resposta do endpoint de listagem de números mudou para o formato Stripe-like com paginação por cursor.

**Antes:**

```json
{
  "numbers": [
    { "alias": "vendas-sp", "phone": "+5511988887777", "status": "active" }
  ]
}
```

**Depois:**

```json
{
  "object": "list",
  "data": [
    {
      "alias": "vendas-sp",
      "phone_number": "+5511988887777",
      "status": "active",
      "number_type": "extra",
      "created_at": "2026-04-01T10:00:00Z"
    }
  ],
  "has_more": false
}
```

**O que fazer:**
- Acesse os números em `data` (antes: `numbers`).
- Atualize referências ao campo `phone` para `phone_number` dentro dos objetos de número.
- Implemente suporte à paginação usando `starting_after` e `has_more` para contas com muitos números.

## Novos recursos nesta versão

Além das mudanças incompatíveis, esta versão introduz novos recursos:

| Recurso | Endpoint | Descrição |
| ------- | -------- | --------- |
| Consulta de mensagem | `GET /v1/messages/{message_id}` | Recupere corpo completo, status e mídia por ID |
| Envio de mídia | `POST /v1/messages` com `content_type` + `media` | Envie imagens, áudios, vídeos, documentos e figurinhas |
| BYO S3 | Dashboard → Settings → Integrations | Armazenamento durável para mídia recebida e enviada |
| Polling de eventos | `GET /v1/events` | Alternativa ou complemento ao webhook para processar eventos |

Consulte as páginas de cada recurso para detalhes de uso.
