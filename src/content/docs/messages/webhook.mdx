O Turbo Notify notifica seu sistema sobre eventos — envio confirmado, entrega, leitura, mensagem recebida, reação, falha — via webhook (HTTP POST). O objeto `data` de cada evento é **idêntico** ao retornado pelo endpoint [`GET /events`](/messages/polling/); a diferença entre os dois canais está apenas no envelope (campos `timestamp` vs `created_at` e a presença do tag `object: "event"` no polling).

## Recebimento dos eventos

Para começar a receber eventos, cadastre a URL do seu webhook no <DashboardLink path="/webhook">Dashboard</DashboardLink>. A URL deve ser pública, acessível via HTTPS e responder com status `2xx` dentro de 5 segundos.

:::tip[Inspecione e aprenda em tempo real]
Antes de plugar o webhook na sua aplicação, abra o <ExternalLink href="https://webhook.turbonotify.com">Webhook Inspector</ExternalLink> e cadastre a URL gerada. O Inspector recebe cada delivery em tempo real (via WebSocket), exibe headers/body completos, valida a assinatura HMAC interativamente e sugere a próxima chamada REST típica para cada tipo de evento (`message.received` → `GET /v1/messages/{id}`).
:::

## Privacidade

Recomendamos que sua URL de webhook não exponha informações sensíveis, como chaves de acesso ou identificadores internos. Use caminhos genéricos e valide a autenticidade da requisição pelo cabeçalho de assinatura.

## Segurança

Toda requisição de webhook enviada pelo Turbo Notify inclui os cabeçalhos abaixo. Utilize a assinatura para garantir que a requisição foi realmente enviada pelo Turbo Notify e não por terceiros.

| Header | Descrição |
|--------|-----------|
| `X-Webhook-Signature` | `sha256=<hex>` — HMAC-SHA256 de `{X-Webhook-Timestamp}.{raw_body}` com seu segredo. |
| `X-Webhook-Timestamp` | Unix epoch (segundos) — utilize para rejeitar deliveries com mais de ~5 minutos (proteção contra replay). |
| `X-Webhook-Attempt` | Número da tentativa atual (começa em 1, incrementa nos retries). |
| `X-Webhook-Max-Attempts` | Total de tentativas que o Turbo Notify fará antes de desistir. |
| `User-Agent` | `TurboNotify-Webhook/1.0` — útil para WAF allowlists. |

O segredo para validação está disponível no <DashboardLink path="/webhook">Dashboard</DashboardLink>, na mesma seção onde você cadastra a URL.

Exemplo de validação da assinatura:

```python title="validacao.py"
import hmac
import hashlib

def verificar_assinatura(payload: bytes, timestamp: str, assinatura: str, segredo: str) -> bool:
    dados_assinados = f"{timestamp}.".encode() + payload
    esperado = "sha256=" + hmac.new(
        segredo.encode(),
        dados_assinados,
        hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(esperado, assinatura)

# Uso:
# timestamp = request.headers["X-Webhook-Timestamp"]
# assinatura = request.headers["X-Webhook-Signature"]
# verificar_assinatura(request.body, timestamp, assinatura, segredo)
```

```js title="validacao.js"
const crypto = require('crypto');

function verificarAssinatura(payload, timestamp, assinatura, segredo) {
  const dadosAssinados = `${timestamp}.${payload}`;
  const esperado = 'sha256=' + crypto
    .createHmac('sha256', segredo)
    .update(dadosAssinados)
    .digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(esperado),
    Buffer.from(assinatura)
  );
}

// Uso:
// const timestamp = req.headers['x-webhook-timestamp'];
// const assinatura = req.headers['x-webhook-signature'];
// verificarAssinatura(req.body, timestamp, assinatura, segredo);
```

## Estrutura do evento

Toda requisição de webhook segue a mesma estrutura base:

| Propriedade  | Tipo      | Descrição                                                                 |
| ------------ | --------- | ------------------------------------------------------------------------- |
| `id`         | `string`  | Identificador único do evento (`evt_…`). Use como chave de idempotência.  |
| `type`       | `string`  | Tipo do evento (ex.: `message.sent`).                                     |
| `timestamp`  | `string`  | Data e hora do evento em ISO 8601 UTC, com sufixo `Z`.                    |
| `request_id` | `string?` | ID de rastreamento da requisição que originou o evento. Omitido quando ausente. |
| `data`       | `object`  | Dados específicos do evento (veja a tabela em [Tipos de evento](#tipos-de-evento)). |

### Exemplo — `message.sent` (outbound)

```json title="webhook-payload.json"
{
  "id": "evt_abc123def456789012345678",
  "type": "message.sent",
  "timestamp": "2026-04-22T10:00:05Z",
  "request_id": "req_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4",
  "data": {
    "message_id": "msg_01993c5a1b24d3e9f01a2b3c4d5e6f70",
    "from": { "number": "5511999990001", "alias": "main" },
    "to":   { "number": "5511999990002" },
    "content_type": "text",
    "preview": "Sua mensagem...",
    "sent_at": "2026-04-22T10:00:05Z"
  }
}
```

`from.alias` é sempre preenchido. Use `"main"` para identificar o número principal ou o alias do número extra (ex.: `"vendas-sp"`).

`content_type` e `preview` espelham `message.received` — `text`,
`image`, `audio`, `video`, `document`, `sticker`, `location`,
`contact` ou `calendar`. Para o body completo (vCard, coordenadas,
descrição do evento, etc.), faça `GET /v1/messages/{message_id}`.

### Exemplo — `message.received` (inbound)

```json title="webhook-payload.json"
{
  "id": "evt_def789abc234567890123456",
  "type": "message.received",
  "timestamp": "2026-04-24T15:32:11Z",
  "data": {
    "message_id": "msg_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4",
    "from": { "number": "5511777777777", "name": "João Silva" },
    "to":   { "number": "5511999990001", "alias": "main" },
    "content_type": "text",
    "preview": "Olá, gostaria de saber...",
    "received_at": "2026-04-24T15:32:11Z"
  }
}
```

`from.name` traz o nome de exibição que o contato configurou no WhatsApp (push name); o campo é omitido quando o contato não definiu nenhum. `to.alias` resolve para `"main"` quando a mensagem chegou no número principal ou para o alias do número extra que recebeu.

### Exemplo — `message.replied` (inbound/outbound)

Disponível desde 2026-04-28. Replies (mensagens que citam outra) são entregues como evento dedicado em vez de `message.sent`/`message.received`. Carrega qualquer `content_type` (text/image/video/audio/document/sticker/location/contact/calendar) — o tipo real do conteúdo é preservado, e a discriminação como reply vem do campo `quoted_message_id` (sempre presente neste evento).

```json title="message.replied (inbound)"
{
  "id": "evt_replied_inbound",
  "type": "message.replied",
  "timestamp": "2026-04-28T10:00:00Z",
  "data": {
    "message_id": "msg_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4",
    "direction": "inbound",
    "from": { "number": "5511777777777", "name": "João Silva" },
    "to":   { "number": "5511999990001", "alias": "main" },
    "content_type": "text",
    "preview": "Sim, é esse pedido mesmo!",
    "replied_at": "2026-04-28T10:00:00Z",
    "quoted_message_id": "msg_99887766554433221100ffeeddccbbaa"
  }
}
```

```json title="message.replied (outbound — enviado via API)"
{
  "id": "evt_replied_outbound",
  "type": "message.replied",
  "timestamp": "2026-04-28T10:01:00Z",
  "request_id": "req_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4",
  "data": {
    "message_id": "msg_b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5",
    "direction": "outbound",
    "from": { "number": "5511999990001", "alias": "main" },
    "to":   { "number": "5511777777777" },
    "content_type": "text",
    "preview": "Sem problemas, vou conferir.",
    "replied_at": "2026-04-28T10:01:00Z",
    "quoted_message_id": "msg_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4"
  }
}
```

Para responder uma mensagem via API, envie `POST /v1/messages` com `reply_to_message_id` apontando para a mensagem citada (qualquer mensagem do chat, inbound ou outbound). Veja [/messages/send/](/messages/send/).

`quoted_message_id` é sempre o id público (`msg_{hex32}`) da mensagem citada. Use `GET /v1/messages/{quoted_message_id}` para recuperar o conteúdo completo da mensagem original.

### Exemplo — `message.delivered` / `message.read` (receipt inbound)

Receipts são sempre **inbound** — o peer A confirma a entrega/leitura de uma mensagem que o seu número (B) enviou. O evento carrega `direction: "inbound"`, party shape inbound (peer no `from`, seu número no `to`) e snapshot do conteúdo original (`content_type` + `preview`) para facilitar debug sem `GET /v1/messages/{id}` extra.

```json title="message.delivered"
{
  "id": "evt_delivered_001",
  "type": "message.delivered",
  "timestamp": "2026-04-28T10:00:30Z",
  "data": {
    "message_id": "msg_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4",
    "direction": "inbound",
    "from": { "number": "5511777777777" },
    "to":   { "number": "5511999990001", "alias": "main" },
    "content_type": "text",
    "preview": "Olá, gostaria de saber...",
    "delivered_at": "2026-04-28T10:00:30Z"
  }
}
```

`preview` fica vazio quando o corpo da mensagem original já foi purgado pelo retention (LGPD); `content_type` é preservado.

### Exemplos por `content_type` (inbound)

O envelope do webhook é idêntico em estrutura para todos os tipos
inbound — apenas `data.content_type` e `data.preview` mudam. Os
trechos abaixo mostram só o objeto `data` para reduzir ruído visual.

Para obter o **body completo** (texto, vCard, coordenadas, descrição
do evento, link da mídia, etc.), faça `GET /v1/messages/{message_id}`.
Os exemplos de body abaixo refletem o `body` retornado por esse
endpoint dentro da janela de retenção.

#### `text` — texto simples

```json title="data — message.received (text)"

  "message_id": "msg_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4",
  "from": { "number": "5511777777777", "name": "João Silva" },
  "to":   { "number": "5511999990001", "alias": "main" },
  "content_type": "text",
```

## Tipos de evento

O catálogo público é fechado — seu endpoint só recebe os 24 eventos listados aqui.

### Mensagens — ciclo de envio

| Tipo                      | Quando é emitido                                                   |
| ------------------------- | ------------------------------------------------------------------ |
| `message.sent`            | Mensagem enviada com sucesso ao WhatsApp (sem citação).            |
| `message.send.failed`     | Falha no envio de uma mensagem regular (sem citação).              |
| `message.delivered`       | Receipt inbound: peer entregou uma mensagem que você enviou.       |
| `message.read`            | Receipt inbound: peer leu uma mensagem que você enviou.            |

### Mensagens — replies (citações)

| Tipo                      | Quando é emitido                                                   |
| ------------------------- | ------------------------------------------------------------------ |
| `message.replied`         | Reply enviado/recebido (qualquer `content_type`, ambas direções).  |
| `message.reply.failed`    | Falha ao enviar um reply (mensagem com `reply_to_message_id`).     |

### Mensagens — recebimento e operações

| Tipo                      | Quando é emitido                                                   |
| ------------------------- | ------------------------------------------------------------------ |
| `message.received`        | Mensagem recebida de um contato (texto, mídia — sem citação).      |
| `message.edited`          | Mensagem foi editada (sua ou do contato).                          |
| `message.edit.failed`     | Falha ao editar uma mensagem.                                      |
| `message.deleted`         | Mensagem foi deletada/revogada (sua ou do contato).                |
| `message.delete.failed`   | Falha ao deletar uma mensagem.                                     |
| `message.reacted`         | Uma reação foi adicionada ou removida.                             |
| `message.react.failed`    | Falha ao reagir a uma mensagem.                                    |

### Números — ciclo de vida

| Tipo                          | Quando é emitido                                                  |
| ----------------------------- | ----------------------------------------------------------------- |
| `number.created`              | Número extra provisionado.                                        |
| `number.connected`            | Número principal ou extra conectado ao WhatsApp.                  |
| `number.disconnected`         | Número desconectado temporariamente.                              |
| `number.connect.failed`       | Falha no pareamento/conexão.                                      |
| `number.revoked`              | Dispositivo desvinculado pelo WhatsApp.                           |
| `number.remove.scheduled`     | Exclusão de número extra agendada (período de carência).          |
| `number.remove.canceled`      | Exclusão agendada foi cancelada.                                  |
| `number.removed`              | Número extra foi removido.                                        |

### Armazenamento e mídia

Eventos relacionados à integração BYO Object Storage (AWS S3, Cloudflare R2, Supabase Storage) e à disponibilidade de mídia recebida.

:::note
Estes eventos são **provider-agnostic**: o payload é o mesmo independentemente do vendor configurado (AWS S3, Cloudflare R2 ou Supabase Storage). O protocolo de transporte é sempre a S3 HTTP API.
:::

:::caution[Mensagens recebidas com mídia — entrega só após resolução do upload]
Para mensagens **inbound com mídia** (imagem, vídeo, áudio, documento, sticker), o webhook `message.received` só é enviado depois que o upload chega a um estado *terminal* para o seu tenant — bytes salvos no seu bucket BYO (`media.storage = "tenant_storage"`), fallback intencional para a URL direta do WhatsApp (`media.storage = "whatsapp_direct"`), ou falha permanente sinalizada por `media.unavailable`.

Em caso de falha transitória de upload (rede, 5xx do storage, etc.), Turbo Notify faz retry em background e só então decide o desfecho. Você nunca verá um `message.received` com `media.storage = "whatsapp_direct"` apenas porque um retry ainda está em andamento.

Se os retries esgotarem ou a janela de 24 h do WhatsApp expirar, **o `message.received` daquela mensagem é intencionalmente suprimido** e você recebe `media.unavailable` no lugar — enviar um webhook de mensagem recebida cujos bytes estão perdidos seria enganoso. Use `media.unavailable` como sinal definitivo de que aquela mídia não pôde ser entregue.
:::

#### Saúde do storage

| Tipo                          | Quando é emitido                                                  |
| ----------------------------- | ----------------------------------------------------------------- |
| `storage.degraded`            | Falhas consecutivas na integração BYO Object Storage abriram o circuito. |
| `storage.recovered`           | Primeira operação bem-sucedida após o circuito ter sido aberto — operação normal restabelecida. |

#### Disponibilidade de mídia

## Mapeamento de operações assíncronas

Operações da API que disparam processamento assíncrono retornam um `request_id`. O mesmo `request_id` aparece no webhook do evento resultante — use para correlacionar:

| Operação                   | Evento de sucesso             | Evento de falha                   |
| -------------------------- | ----------------------------- | --------------------------------- |
| Enviar mensagem            | `message.sent`                | `message.send.failed`             |
| Responder mensagem         | `message.sent`                | `message.send.failed`             |
| Editar mensagem            | `message.edited`              | `message.edit.failed`             |
| Deletar mensagem           | `message.deleted`             | `message.delete.failed`           |
| Reagir a uma mensagem      | `message.reacted`             | `message.react.failed`            |
| Conectar número extra      | `number.connected`            | `number.connect.failed`           |
| Agendar remoção de número  | `number.remove.scheduled`     | —                                 |
| Cancelar remoção agendada  | `number.remove.canceled`      | —                                 |

## Tentativas de entrega

Se seu endpoint não responder com status `2xx` dentro de 5 segundos, o Turbo Notify refará a tentativa com backoff exponencial:

| Tentativa | Intervalo aproximado |
| :-------: | -------------------- |
| 1         | Imediata             |
| 2         | 12 segundos          |
| 3         | 24 segundos          |
| 4         | 48 segundos          |
| 5         | 96 segundos          |
| 6         | 192 segundos         |

Após esgotar as tentativas, o evento é marcado como `dead_letter` e descartado.

## Garantia de entrega

O Turbo Notify adota o modelo **at-least-once**. Em situações excepcionais, o mesmo evento pode ser entregue mais de uma vez. Use o campo `id` para implementar idempotência e evitar processamento duplicado.

```js title="dedup.js"
const seen = new Set();

function handleEvent(event) {
  if (seen.has(event.id)) {
    return; // já processado
  }
  seen.add(event.id);
  // ... processar
}
```

Em produção, persista o `id` em uma tabela ou cache com TTL de ~72 h.

## Alternativa: polling

Se preferir consumir eventos sob demanda em vez de expor um endpoint público, utilize o [polling via `GET /events`](/messages/polling/). O objeto `data` é o mesmo do webhook.
