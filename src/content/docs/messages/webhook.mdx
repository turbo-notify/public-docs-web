O Turbo Notify utiliza webhooks para notificar seu sistema sobre eventos importantes, como a confirmação de envio de uma mensagem, a entrega ao destinatário ou a ocorrência de uma falha. Dessa forma, você não precisa consultar a API repetidamente para acompanhar o andamento das operações.

## Recebimento dos eventos

Para começar a receber eventos, cadastre a URL do seu webhook no <DashboardLink path="/webhook">Dashboard</DashboardLink>. A URL deve ser pública, acessível via HTTPS e responder com status `200` dentro de 5 segundos.

## Privacidade

Recomendamos que sua URL de webhook não exponha informações sensíveis, como chaves de acesso ou identificadores internos. Use caminhos genéricos e valide a autenticidade da requisição pelo cabeçalho de assinatura.

## Segurança

Toda requisição de webhook enviada pelo Turbo Notify inclui os cabeçalhos `X-Webhook-Signature` e `X-Webhook-Timestamp` com uma assinatura HMAC SHA-256. Utilize essa assinatura para garantir que a requisição foi realmente enviada pelo Turbo Notify e não por terceiros.

O segredo para validação está disponível no <DashboardLink path="/webhook">Dashboard</DashboardLink>, na mesma seção onde você cadastra a URL.

Exemplo de validação da assinatura:

```python title="validacao.py"
import hmac
import hashlib

def verificar_assinatura(payload: bytes, timestamp: str, assinatura: str, segredo: str) -> bool:
    dados_assinados = f"{timestamp}.".encode() + payload
    esperado = "sha256=" + hmac.new(
        segredo.encode(),
        dados_assinados,
        hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(esperado, assinatura)

# Uso:
# timestamp = request.headers["X-Webhook-Timestamp"]
# assinatura = request.headers["X-Webhook-Signature"]
# verificar_assinatura(request.body, timestamp, assinatura, segredo)
```

```js title="validacao.js"
const crypto = require('crypto');

function verificarAssinatura(payload, timestamp, assinatura, segredo) {
  const dadosAssinados = `${timestamp}.${payload}`;
  const esperado = 'sha256=' + crypto
    .createHmac('sha256', segredo)
    .update(dadosAssinados)
    .digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(esperado),
    Buffer.from(assinatura)
  );
}

// Uso:
// const timestamp = req.headers['x-webhook-timestamp'];
// const assinatura = req.headers['x-webhook-signature'];
// verificarAssinatura(req.body, timestamp, assinatura, segredo);
```

## Estrutura do evento

Toda requisição de webhook segue a mesma estrutura base:

| Propriedade | Tipo     | Descrição                                          |
| ----------- | -------- | -------------------------------------------------- |
| `id`        | `string` | Identificador único do evento.                     |
| `type`      | `string` | Tipo do evento (ex: `message.sent`).               |
| `timestamp` | `string` | Data e hora do evento em formato ISO 8601.         |
| `data`      | `object` | Dados específicos do evento.                       |

### Exemplo

```json title="webhook-payload.json"
{
  "id": "evt_abc123",
  "type": "message.sent",
  "timestamp": "2025-07-16T10:00:02Z",
  "data": {
    "message_id": "msg_abc123",
    "to": "+5511999999999",
    "status": "sent"
  }
}
```

## Tipos de evento

| Evento                          | Descrição                                                           |
| ------------------------------- | ------------------------------------------------------------------- |
| `message.sent`                  | A mensagem foi enviada ao WhatsApp com sucesso.                     |
| `message.delivered`             | A mensagem foi entregue ao dispositivo do destinatário.             |
| `message.read`                  | A mensagem foi lida pelo destinatário.                              |
| `message.failed`                | O envio da mensagem falhou.                                         |
| `message.received`              | Uma mensagem foi recebida de um contato.                            |
| `message.reaction`              | Uma reação foi adicionada ou removida de uma mensagem.              |
| `session.connected`             | A sessão do WhatsApp foi conectada com sucesso.                     |
| `session.disconnected`          | A sessão do WhatsApp foi desconectada.                              |
| `extra_number.paired`           | Um número extra foi pareado com sucesso.                            |
| `extra_number.disconnected`     | Um número extra foi desconectado.                                   |
| `extra_number.removed`          | Um número extra foi removido da conta.                              |
| `extra_number.pairing_failed`   | O pareamento de um número extra falhou.                             |
| `extra_number.pairing_expired`  | O código de pareamento de um número extra expirou.                  |

## Mapeamento de operações assíncronas

Algumas operações da API retornam um `operation_id` para rastreamento. Veja como cada operação se relaciona com os eventos de webhook:

| Operação                   | Evento de sucesso             | Evento de falha                   |
| -------------------------- | ----------------------------- | --------------------------------- |
| Enviar mensagem            | `message.sent`                | `message.failed`                  |
| Responder mensagem         | `message.sent`                | `message.failed`                  |
| Parear número extra        | `extra_number.paired`         | `extra_number.pairing_failed`     |
| Remover número extra       | `extra_number.removed`        | -                                 |

## Tentativas de entrega

Caso o seu endpoint não responda com status `200` dentro de 5 segundos, o Turbo Notify fará novas tentativas com backoff exponencial:

| Tentativa | Intervalo aproximado |
| :-------: | -------------------- |
| 1         | Imediata             |
| 2         | 30 segundos          |
| 3         | 2 minutos            |
| 4         | 10 minutos           |
| 5         | 1 hora               |

Após 5 tentativas sem sucesso, o evento será descartado.

## Garantia de entrega

O Turbo Notify adota o modelo **at-least-once delivery**. Isso significa que, em situações excepcionais, o mesmo evento pode ser entregue mais de uma vez. Utilize o campo `id` do evento para implementar idempotência no seu sistema e evitar processamento duplicado.
